You are triaging a production incident. Your job is to analyze the error, identify the root cause, check for regressions in recent commits, and propose a fix with tests.

## Production Error

```
{ERROR_TEXT}
```

## Instructions

### Phase 1 — Parse and Understand the Error

1. **Parse the error**: Identify the exception type, error message, and stack trace frames. Determine which module, function, and line triggered the error.
2. **Read the project's CLAUDE.md** (if it exists) for coding conventions and project structure.
3. **Locate the affected code**: Use Read, Glob, and Grep to find the file(s) and function(s) mentioned in the error. Read them to understand the current behavior.

### Phase 2 — Investigate Root Cause

4. **Analyze the code path**: Trace the execution flow that leads to the error. Identify what input, state, or condition triggers it.
5. **Check recent commits for regression**: Run `git log --oneline -20 -- <affected-files>` to see if recent changes could have introduced the bug. If a suspect commit is found, examine its diff with `git show <commit-hash>`.
6. **Determine root cause**: Is this a regression from a recent change, a latent bug, or a new edge case? State your conclusion clearly.

### Phase 3 — Write Tests

7. **Write tests that reproduce the issue** before fixing it. Follow existing test patterns (pytest, `tests/test_*.py`). The tests should FAIL before the fix is applied. Tests should verify observable behavior (return values, exceptions, state changes) — not assert that specific internal functions were called.
8. If the issue cannot be reproduced in tests (infrastructure, config, external dependency), note why and skip this step.

### Phase 4 — Fix

9. **Create a branch**: `{BRANCH_PREFIX}incident-{TIMESTAMP}`. If already on a feature branch, stay on it.
10. **Implement the fix.** Edit the minimal set of files needed. Follow project conventions strictly.
11. **Run tests** to verify the fix. Ensure the reproduction test now passes and no existing tests break.
12. **Commit** with a clear message: `fix: <what was fixed>` — include root cause in the commit body.

### Phase 5 — Submit Pull Request

13. **Push the branch** to origin:
    ```bash
    git push -u origin HEAD
    ```

14. **Create a draft pull request** using `gh`:
    ```bash
    gh pr create --draft --title "fix: <concise title>" --body "$(cat <<'EOF'
    ## Summary

    [Root cause analysis and what was fixed — 2-3 sentences]

    ## Incident Details

    ```
    [First 10 lines of the original error]
    ```

    ## Root Cause

    [What caused the error and why]

    ## Changes

    - [Key change 1]
    - [Key change 2]

    ## Test plan

    - [How the fix was verified]

    ---
    *Generated by Kōan /incident*
    EOF
    )"
    ```
    - If the local repo is a fork, submit to the upstream repository:
      ```bash
      gh pr create --draft --repo <upstream-owner>/<repo> --head <fork-owner>:<branch> --title "..." --body "..."
      ```
    - PRs are **always draft**. Never create a non-draft PR.

## Output Format

When you are done, output a structured summary in this exact format:

```
INCIDENT_SUMMARY_START
root_cause: <one-line root cause description>
culprit_commit: <commit hash if regression, "none" otherwise>
fix_description: <one-line description of the fix>
affected_files: <comma-separated list of files changed>
pr_url: <PR URL if created, "none" otherwise>
INCIDENT_SUMMARY_END
```

## Rules

- **Minimal changes.** Fix the incident, don't refactor unrelated code.
- **Never commit to main.** Always work on the feature branch.
- **Test before commit.** Never commit code that breaks tests.
- **Be surgical.** Smallest change that resolves the production error.
- **State assumptions.** If you're uncertain about the root cause, say so explicitly and explain what additional information would help.
- **Escalate when stuck.** If you cannot identify the root cause or a fix, clearly state what you found and what you couldn't determine.
